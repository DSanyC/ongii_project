# Ongii 앱 완성을 위한 다음 진행 사항

## 목표
현재 구조를 유지하면서 `실행 가능한 MVP`를 먼저 완성하고, 이후 안정화/고도화 단계로 확장한다.

## 1) 이번 주 최우선 (MVP 완성)
- 공통 네비게이션/라우팅 점검: `home`, `calendar`, `feed`, `chat`, `my` 화면 이동 흐름 고정
- 더미 데이터 연결: 각 화면이 빈 화면이 아니라 테스트 가능한 리스트/상세 상태를 보여주도록 정리
- 핵심 유즈케이스 1개씩 완성
- `calendar`: 일정 등록/수정/삭제
- `feed`: 게시글 작성/목록/상세
- `chat`: 대화방 목록/메시지 송수신 UI
- `my`: 가족 구성원 전환/프로필 수정
- 예외 상태 처리: 로딩/빈 상태/에러 UI 공통 패턴 적용

## 2) 다음 단계 (기능 안정화)
- 상태관리 일원화 (현재 방식 유지 또는 Provider/Riverpod 중 하나로 고정)
- 데이터 계층 정리
- model, repository, service 레이어 분리
- API 붙이기 전 mock repository 인터페이스 확정
- 입력 검증 강화: 폼 validation, 에러 메시지 일관성
- 디자인 시스템 최소화
- 색상, 타이포, 버튼/카드/폼 컴포넌트 규격 통일

## 3) 품질 확보 (출시 전 필수)
- 테스트 작성
- 위젯 테스트: 주요 컴포넌트 렌더링/동작
- 통합 테스트: 일정 생성, 피드 작성, 채팅 입력 플로우
- 성능/안정성
- 초기 진입 속도, 스크롤 끊김, 이미지 로딩 점검
- 크래시 방지를 위한 null/비동기 예외 처리 보강
- QA 체크리스트 작성
- 네트워크 끊김, 권한 거부, 빈 데이터, 오래된 앱 버전 대응

## 4) 배포 준비
- 앱 아이콘/스플래시/앱 이름 확정
- Android 서명키 및 빌드 타입(dev/staging/prod) 분리
- 개인정보처리방침/서비스 소개 문서 준비
- 내부 테스트 배포(Test track) 후 피드백 반영

## 권장 실행 순서 (간단)
1. 화면별 핵심 유즈케이스 1개씩 완성 (MVP)
2. 상태관리 + 데이터 계층 정리
3. 테스트 및 예외 처리 강화
4. 내부 배포 후 수정
